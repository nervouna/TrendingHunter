# Trending Product Analyst — PRD

## 1. Overview

A standalone CLI tool that fetches trending open-source projects from multiple sources, filters them through a signal gate, and generates structured analysis reports to a local markdown knowledge base.

## 2. User Stories

### US-1: Fetch Trending Projects
**As a** technical founder
**I want to** fetch trending repositories from GitHub, Product Hunt, and Hacker News in one command
**So that** I don't need to browse each source manually

**Status:** GitHub, Product Hunt, and Hacker News implemented.

### US-2: Automatic Signal Filtering
**As a** user
**I want** the system to discard low-signal projects before analysis
**So that** I only spend attention on genuinely noteworthy things

**Status:** Implemented.

**Acceptance Criteria:**
- Filters by star growth velocity (not raw count)
- Prioritizes young repos over mature ones gaining stars
- Weighs first-time contributor count as a novelty signal
- Retains all filtering metrics for later reference

### US-3: Generate Structured Reports
**As a** user
**I want** each analyzed project to produce a standardized report
**So that** I can compare trends across time using a consistent framework

**Status:** Implemented.

**Acceptance Criteria:**
- Report follows the 11-section template (TL;DR through Open Questions)
- Filename pattern: `{date}-{source}-{project-name}.md`
- Saved to `reports/` directory

### US-4: Three-Stage LLM Pipeline
**As a** user
**I want** reports generated by a cheap model, audited by a flagship model, then rewritten for clean prose
**So that** I get high quality at controlled cost

**Status:** Implemented.

**Acceptance Criteria:**
- Draft stage uses a low-cost LLM (configurable, default MiniMax-M2.7-highspeed)
- Audit stage uses a flagship LLM (configurable, default deepseek-reasoner) with Tavily tools for fact-checking
- Rewrite stage uses a low-cost LLM (same as draft) to polish the audited report into clean prose
- All three stages are independently configurable via `config.yaml`

### US-5: Configurable Scheduling
**As a** user
**I want to** run the pipeline on a schedule via cron or a runner script
**So that** the knowledge base stays current without manual intervention

**Status:** Not implemented. Manual `th run` only; external cron not yet set up.

### US-6: Knowledge Base Search
**As a** user
**I want to** search past reports by keyword, date, or source
**So that** I can spot meta-patterns and compare trends over time

**Status:** Implemented.

### US-7: Cost Control
**As a** user
**I want to** see token usage and estimated cost per run
**So that** I can manage LLM spending

**Status:** Implemented.

### US-8: Source Configuration
**As a** user
**I want to** enable/disable data sources and adjust signal gate thresholds via config
**So that** I can tune the pipeline without code changes

**Status:** Implemented.

## 4. Data Model

```python
Project:
  name: str
  source: enum(github, producthunt, hackernews)
  url: str
  fetched_at: datetime
  stars: int
  star_velocity: float          # stars/day
  repo_age_days: int
  first_time_contributors: int
  description: str
  readme_excerpt: str

Report:
  project: Project
  generated_at: datetime
  draft_model: str
  audit_model: str
  rewrite_model: str
  token_usage: {draft: {input: int, output: int}, audit: {input: int, output: int}, rewrite: {input: int, output: int}}
  sections: dict[str, str]      # section_name → content
  file_path: str
```

## 5. Report Template

Each report follows 11 sections:

1. **TL;DR** — Two sentences: what this is and why it matters.
2. **What & Why** — The project's own narrative. What problem does it claim to solve?
3. **Why Now** — Why couldn't this exist two years ago? What shifted in the ecosystem (cost, infrastructure, demand) that makes this viable today?
4. **Technology Wave** — Which macro trend does this ride? What's the underlying force?
5. **Supply & Demand** — Who's building it, who's using it, and what's the demand signal?
6. **Product Analysis** — For what, for whom, business model.
7. **Creativity & Differentiation** — What's genuinely new vs. incremental? What existing solutions did the author reject and why?
8. **Competitive Landscape** — Who else solves this? Why might someone choose this instead?
9. **Community Signals** — Contributor mix, discussion quality, velocity indicators.
10. **Signal Assessment** — Real trend or temporary hype? Confidence level.
11. **Open Questions** — What's unclear. What to investigate next.

## 6. Non-Functional Requirements

- **Language**: Python 3.11+
- **Dependencies**: click, httpx, pyyaml, pydantic, anthropic
- **No web server** — standalone scripts only
- **Idempotent runs** — re-running with same config should not duplicate reports (not yet implemented)
- **All config via file** — no hardcoded keys, models, or thresholds
- **Retry logic** — exponential backoff on LLM API errors (connection, rate limit, timeout)
- **Web research** — Tavily API for search and content extraction during audit stage
